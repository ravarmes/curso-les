Objetivos

Objetivos da etapa

1. Separação de responsabilidades

O aluno deve perceber que uma API organizada não coloca tudo dentro de um único arquivo. Cada camada resolve um tipo de problema.

2. Fluxo completo da requisição

É importante mostrar como o dado sai do frontend, entra pela rota, percorre controller, service, model e retorna em JSON.

3. Organização escalável

Quando surgem novos cadastros, a estrutura já está pronta: normalmente adicionamos rota, controller, service e model.

4. Reuso de padrão

Os pacotes do projeto seguem um padrão repetível. Isso facilita explicar que aprender um recurso ajuda a entender todos os outros.

Ideia central

O projeto não é apenas um conjunto de arquivos. Ele é uma linha de montagem da informação: cada pacote recebe o dado, trata uma parte da responsabilidade e repassa para a próxima camada.

Estrutura de Pastas

Estrutura do src

controllers

Camada de entrada HTTP

Recebe a chamada da rota, aciona o service correto e devolve a resposta JSON ao cliente.

services

Camada de regras e operações

Implementa regras de negócio, valida dependências, cria transações e conversa com os models.

models

Camada de mapeamento do banco

Define campos, tipos, validações, nomes de tabelas e relacionamentos entre entidades.

Infraestrutura de banco

Centraliza configuração do Sequelize, conexão, sincronização e carga inicial de dados.

routes.js

Mapa da API

Expõe endpoints como /ufs, /filmes e /fitas, apontando para os controllers.

_middleware

Tratamento centralizado de erros

Converte falhas em respostas HTTP padronizadas, evitando repetição de tratamento em cada classe.

Fluxo da Requisição

Fluxo da requisição

Passo 1

Frontend

Envia uma requisição HTTP para a API, por exemplo POST /ufs.

Passo 2

Route

Identifica qual controller deve atender aquele endpoint.

Passo 3

Controller

Recebe req e res, delega ao service e monta a resposta.

Passo 4

Service

Executa regras de negócio, valida dados e chama os models necessários.

Passo 5

Model

Usa o Sequelize para acessar tabela, validar campos e percorrer relacionamentos.

Passo 6

Resposta JSON

O resultado volta pela mesma trilha até o cliente, já serializado em JSON.

Ideia principal

Em uma arquitetura em camadas, cada pacote assume uma função específica. Isso deixa a aplicação mais organizada, facilita manutenção e reduz a mistura entre regras de negócio, acesso ao banco e resposta HTTP.

Controller

Função do controller

O que ele faz

  • Recebe a requisição através das rotas do Express.
  • Chama o service correspondente à ação desejada.
  • Transforma o retorno em JSON com res.json(...).
  • Encaminha erros com catch(next) para o middleware global.
  • Permanece enxuto, sem colocar regra de negócio complexa.

Exemplo real: UfController

UfController.js 
class UfController {
  static async create(req, res, next) {
    UfService.create(req)
      .then(obj => res.json(obj))
      .catch(next);
  }
}

Service

Função do service

O que ele faz

  • Extrai dados de req.params e req.body.
  • Aplica regras de negócio, como verificar dependências obrigatórias.
  • Coordena o acesso ao model sem acoplar essa lógica ao controller.
  • Executa o CRUD com regras de busca, criação, alteração e remoção.
  • Recarrega dados quando necessário para devolver o registro persistido.

Exemplo real: UfService

UfService.js 
static async create(req) {
  const { sigla, nome } = req.body;
  const obj = await Uf.create({ sigla, nome });
  return await Uf.findByPk(obj.id, {
    include: { all: true, nested: true }
  });
}

Model

Função do model

O que ele faz

  • Mapeia a tabela com nome do model e nome físico da tabela.
  • Declara atributos com tipos Sequelize, como STRING, INTEGER e DATEONLY.
  • Aplica validações como notEmpty, len, isDate e expressões regulares.
  • Define relacionamentos quando a entidade se conecta a outras tabelas.
  • Representa a entidade que será persistida no banco.

Exemplo real: Uf

Uf.js 
class Uf extends Model {
  static init(sequelize) {
    super.init({
      sigla: {
        type: DataTypes.STRING,
        allowNull: false
      },
      nome: {
        type: DataTypes.STRING,
        allowNull: false
      }
    }, { sequelize, modelName: 'uf', tableName: 'ufs' })
  }
}

Boas Práticas

Como ler a arquitetura

Observe a rota

Ela mostra qual endpoint foi criado e qual controller será acionado quando a requisição chegar.

Leia o controller

Ele normalmente é enxuto e serve para encaminhar a requisição e devolver a resposta ao cliente.

Estude o service

É nele que aparecem as ações do CRUD, as validações e a conversa com o model.

Entenda o model

O model revela como a entidade é salva no banco, quais campos existem e quais validações são aplicadas.

Padrão do projeto

Se uma nova entidade for criada amanhã, o caminho de implementação continua parecido: model, service, controller e rota. Esse padrão torna o backend mais previsível e mais fácil de manter.

Resumo final

Neste projeto, routes mapeiam a API, controllers conversam com HTTP, services concentram regras e models representam o banco com Sequelize. Quando cada camada respeita sua função, o backend fica mais organizado, reutilizável e fácil de expandir.
